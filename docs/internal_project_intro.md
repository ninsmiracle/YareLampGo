# lampgo 对内项目介绍文档

> 面向产品、市场、公关、媒体 briefing、生态合作与内部评审的项目介绍底稿。

## 1. 项目名称

**lampgo**

中文可对外解释为：**桌面具身智能台灯机器人运行时**。

项目面向 LeLamp 5-DOF 桌面机械臂/智能台灯形态，提供一套运行在本地电脑上的机器人运行时系统，统一调度运动、表情、视觉、语音、多模态大模型和上层 Agent 能力。

## 2. 一句话定义

**lampgo 是一个让桌面台灯机器人具备自然语言交互、仿生动作表达、多模态感知和 AI Agent 扩展能力的本地运行时系统。**

更偏产品化的说法：

**它把一台 5 自由度机械台灯，从“可控硬件”升级为“能听、能看、能动、能表达、能被 AI Agent 调度的桌面具身智能伙伴”。**

## 3. 为什么做这个项目

桌面机器人/智能硬件常见的问题不是单点能力缺失，而是能力之间没有被稳定地串起来：

- 硬件能动，但缺少安全、平滑、可复用的运动控制运行时。
- 大模型能理解语言，但很难可靠地映射到真实世界的电机、LED、摄像头和语音设备。
- Demo 能跑通一次，但缺少录制、回放、配置、调试、Web UI、无硬件降级等工程化能力。
- 简单指令需要快速响应，复杂任务又需要长链路推理、代码生成、用户确认和生态工具协同。
- 台灯机器人作为“有身体的 AI”，不只需要回答问题，还需要有动作、表情、视线、语音和环境感知形成完整体验。

lampgo 的目标是补齐这条链路：在底层把真实硬件控制做稳，在中层把动作、表情、语音、视觉抽象成安全技能，在上层通过大模型和 OpenClaw 等 Agent 生态把自然语言、多模态输入和复杂任务连接起来。

## 4. 目标用户是谁

### 主要用户

- **AI 硬件原型团队**：需要快速验证桌面具身智能、机器人交互、智能台灯形态和 Agent 硬件协同体验。
- **产品经理/体验设计团队**：需要基于真实硬件观察用户如何与“有身体的 AI”互动，并沉淀场景、话术和交互范式。
- **算法与大模型应用团队**：需要一个可调用真实执行器、摄像头、麦克风和表情系统的端侧实验平台。
- **开发者和创作者**：希望通过自然语言、录制、脚本或 Agent 快速制作自定义动作和互动玩法。

### 间接受众

- **市场、公关、媒体沟通对象**：可用它展示“小米 AI + 具身智能 + 桌面机器人”的可感知 Demo。
- **生态合作伙伴**：可通过插件、技能和 HTTP 接口把 lampgo 接入更大的 Agent 工作流。
- **内部评审/管理层**：可用它评估桌面智能硬件从单点 Demo 走向可复用平台的技术路线。

## 5. 产品白皮书素材

### 5.1 功能特点

- **自然语言控制台灯机器人**：用户可以通过文字、语音或 Web UI 让台灯点头、摇头、跳舞、看向某个方向、切换表情或回到安全位。
- **多模态感知与交互**：支持摄像头抓帧、语音输入、语音播报、人体存在检测等能力，形成“看、听、说、动”的闭环。
- **仿生动作表达**：内置点头、摇头、摆动、舞蹈、注视、呼吸感等动作，并支持录制和回放用户自定义动作。
- **LED 表情系统**：通过 ESP32 驱动 LED 表情，支持 30+ 预设表情，让机器人具备情绪表达能力。
- **安全优先的运动控制**：所有动作经过安全内核校验，包含关节限位、速度限制、加速度限制、急停和串口健康检查。
- **Web UI 一体化控制台**：提供聊天、状态、设置、录制、表情和技能面板，降低调试和演示门槛。
- **OpenClaw/Agent 生态集成**：复杂任务可升级到外部 Agent，由 Agent 编排动作、调用工具、生成新技能或与用户确认。
- **无硬件开发模式**：缺少设备时可进入 no-hw 模式，便于前端、Agent、配置和意图路由开发。

### 5.2 核心卖点

- **不是普通聊天机器人，而是有身体反馈的 AI**：用户的输入会转化为动作、表情、视线和语音反馈。
- **简单任务快，复杂任务强**：关键词命中可毫秒级路由，复杂任务交给多轮 LLM 工具调用或 OpenClaw 长链路处理。
- **从 Demo 到运行时**：项目不是一次性脚本，而是包含配置、守护进程、IPC、Web UI、安全内核、技能系统和插件生态的完整运行时。
- **动作可被沉淀和复用**：用户录制或 Agent 生成的动作可以保存成技能，后续通过自然语言或 UI 直接调用。
- **真实硬件安全可控**：所有电机命令都通过统一 MotionRuntime 和 SafetyKernel，不向上层暴露危险的裸关节控制。
- **适合宣发演示**：台灯形态天然适合桌面场景，动作和表情具备可拍摄、可传播、可理解的视觉表现力。

### 5.3 应用场景

- **桌面 AI 伙伴**：通过语音和动作陪伴用户，执行问候、提醒、情绪表达和轻量互动。
- **AI 硬件 Demo 展示**：在发布会、展厅、媒体 briefing 中展示大模型如何驱动物理设备。
- **创作者动作工作台**：用户手动录制动作，或描述动作后由 Agent 生成，再通过台灯回放。
- **办公桌智能入口**：作为桌面上的具身 Agent，承接提醒、状态反馈、信息播报、轻量桌面控制等任务。
- **具身智能研发平台**：验证大模型工具调用、视觉伺服、语音闭环、动作规划、安全控制和 Agent 协作。
- **智能硬件生态样板**：展示软件运行时如何把电机、LED、摄像头、麦克风和大模型能力组合成可扩展产品。

## 6. 核心使用场景

### 场景一：自然语言控制

用户输入“点个头”“做个害羞的表情”“跳个舞”，系统通过意图路由匹配到对应技能，驱动机械臂和 LED 表情完成反馈。

适合展示：

- 大模型/意图路由到真实硬件的闭环。
- 低延迟、强反馈的桌面机器人交互。
- 动作和表情带来的“有生命感”体验。

### 场景二：语音对话与身体反馈

用户直接对台灯说话，系统通过 VAD、语音输入、多模态模型和 TTS 完成语音闭环，同时让台灯做动作或表情反馈。

适合展示：

- AI 助手从屏幕走向物理桌面的体验。
- 语音、动作、表情共同组成的多通道交互。
- MiMo 语音和多模态能力在智能硬件上的落地。

### 场景三：录制和回放动作

用户手动摆动机械臂录制一段动作，命名后可通过 Web UI、CLI、自然语言或 Agent 再次触发。

适合展示：

- 非工程用户也能制作机器人动作。
- 动作资产可沉淀为可复用技能。
- 台灯从固定程序变成可学习、可扩展的表达载体。

### 场景四：复杂任务升级到 Agent

当用户提出“帮我设计一个开箱视频打光动作”“每天早上做一个叫醒动作”“帮我生成一个新的欢迎动作”等复杂需求时，本地路由可升级到 OpenClaw，由 Agent 进行多步规划、工具调用、代码/技能生成和用户确认。

适合展示：

- 桌面硬件成为 Agent 的执行终端。
- 复杂任务从自然语言到动作资产的自动化生成。
- Agent 不只是回答问题，而是能创建和改造设备能力。

### 场景五：视觉感知与注视/反应

系统可通过摄像头抓取桌面或用户画面，结合多模态模型进行理解，再触发看向、扫描、问询、表情或动作反馈。

适合展示：

- 多模态大模型与实体设备联动。
- “看到用户/物体后做出反应”的具身智能体验。
- 视觉伺服、桌面场景理解和人机互动探索。

## 7. 核心功能

### 7.1 交互入口

- **Web UI**：默认本地 Web 控制台，支持聊天、录制、配置、状态查看和表情/技能触发。
- **CLI**：支持 run、invoke、text、move、play、record、status、detect、estop、calibrate 等命令。
- **语音循环**：支持麦克风输入、VAD 检测、模型理解、TTS 播报。
- **HTTP/IPC 接口**：供插件、Agent 和其他本地程序调用。
- **OpenClaw 插件**：让外部 Agent 可查询状态、控制动作、读取摄像头、保存技能和向用户提问。

### 7.2 机器人动作能力

- 5 自由度机械臂关节控制。
- 安全位返回、急停、单关节移动。
- 点头、摇头、看向、舞蹈、闲置摆动等参数化动作。
- CSV 轨迹录制、平滑、压缩和回放。
- 不同风格动作表达，如 gentle、confident、curious 等。

### 7.3 表情与反馈能力

- ESP32 LED 表情控制。
- 30+ 预设表情模式。
- 可与动作、语音、任务状态联动。
- 可用于思考中、等待用户、完成、失败等状态提示。

### 7.4 感知与语音能力

- 摄像头抓帧和 base64 图像输入。
- USB 摄像头或 ESP32 无线摄像头接入。
- 本地麦克风或 ESP32 麦克风接入。
- VAD 语音活动检测。
- MiMo STT/音频输入转写。
- MiMo TTS 流式语音合成，Edge TTS 作为回退方案。

### 7.5 Agent 与技能扩展

- 统一 SkillRegistry 管理内置技能和用户技能。
- LLM 自动从技能注册表生成工具调用 schema。
- OpenClaw 可通过插件调用 lampgo 的动作、表情、摄像头、记忆和用户问询能力。
- 支持运行时热加载用户动作和组合技能。

## 8. 技术亮点

### 8.1 级联式意图路由

系统采用三层降级链路：

1. **关键词匹配**：简单命令直接命中，延迟低，适合高频动作和表情。
2. **LLM Agent Loop**：关键词未命中时，进入多轮工具调用，让模型决定调用哪个技能。
3. **OpenClaw 升级**：本地模型判断任务过复杂时，交给外部 Agent 进行长链路推理、编排和确认。

这套设计兼顾了响应速度和任务上限：简单交互不会被大模型链路拖慢，复杂需求也不会被固定技能表卡死。

### 8.2 安全内核 + 运动运行时

所有物理动作都通过 MotionRuntime 和 SafetyKernel：

- 控制线程独立运行，减少 asyncio 调度抖动对电机控制的影响。
- 50Hz 控制循环提供连续、稳定的动作输出。
- 支持梯形速度曲线、弹簧/阻尼式平滑、轨迹帧回放等运动范式。
- 每一帧硬件命令都经过关节限位、速度上限、加速度上限和急停状态检查。
- 上层只能调用技能，不能绕过安全内核直接写电机。

### 8.3 目标驱动与轨迹驱动双范式

项目区分两类运动：

- **目标驱动**：只知道终点，如“看向左边”“回安全位”，由系统规划中间路径。
- **轨迹驱动**：已知完整帧序列，如录制动作和参数化舞蹈，系统按帧稳定回放。

这避免了把录制轨迹拆成大量 move_to 导致的顿挫问题，是项目从“能动”走向“动得自然”的关键工程改进。

### 8.4 单进程运行时架构

lampgo 采用单进程 Python 守护进程，内部拆分为 core、skills、perception、voice、bridge、web 等模块：

- 降低部署和调试复杂度。
- CLI、Web UI、IPC、OpenClaw 共享同一套技能执行器和安全内核。
- 事件总线把技能开始、结束、安全触发、状态变化推送给 WebSocket 和 UI。
- 缺少硬件时可进入 no-hw 模式，支持纯软件调试。

### 8.5 Web UI 与 onboard 工程化能力

项目不仅有硬件控制代码，还包含首次配置和日常操作能力：

- onboard 引导用户检查环境、配置硬件、设置模型和导入人设。
- Web UI 支持模型、硬件、语音、运动和安全参数配置。
- 状态胶囊展示摄像头、麦克风、电机等设备状态。
- 录制、回放、技能和表情可在浏览器中直接操作。

### 8.6 OpenClaw/插件生态集成

lampgo 将台灯能力暴露为 Agent 可调用的工具：

- 查询设备状态。
- 控制关节、播放录制、设置 LED 表情。
- 拍摄摄像头画面并返回给模型分析。
- 保存新录制动作并注册自然语言别名。
- 通过 TTS 和 Web UI 向用户提问并等待回复。

这使台灯机器人从单体设备变成 Agent 工作流中的具身执行端。

## 9. 依赖的小米 AI/模型/软件能力

### 9.1 明确依赖或优先适配的能力

- **MiMo 大模型 API**  
  项目通过 OpenAI-compatible API 接入 MiMo，用于自然语言理解、多轮工具调用、复杂意图处理和多模态输入。

- **mimo-v2.5 / mimo-v2-omni**  
  用于文本、图像、音频等多模态交互。代码中已支持将语音音频或摄像头画面送入模型链路，由模型理解后调用机器人技能。

- **MiMo STT / 音频输入能力**  
  语音转写模块通过 MiMo chat completions 的 input_audio 能力，把 PCM/WAV 音频转成文本或直接进入 Agent Loop。

- **MiMo-V2.5-TTS**  
  用于流式语音合成，输出 24kHz PCM16LE 音频，可边生成边播放，支撑更自然的语音反馈。

- **MiMo 联网搜索能力**  
  配置中支持 web_search_enabled、web_search_force、web_search_limit 等参数，可在模型支持时让台灯回答天气、本地信息或实时信息类问题。

- **OpenAI-compatible / Anthropic-compatible Provider 适配能力**  
  项目已处理不同 provider 兼容路径和参数差异，降低模型替换和内部平台迁移成本。

### 9.2 已集成的软件生态能力

- **OpenClaw 插件与技能生态**  
  lampgo 可作为 OpenClaw 的硬件插件运行，把动作、表情、摄像头、记忆和用户问询能力暴露给 Agent。复杂任务可以由 OpenClaw 进行长链路推理、工具调用、技能生成和用户确认。

- **本地 Web Gateway 能力**  
  Starlette Web 服务同时提供 REST API、WebSocket 实时状态和零构建静态 Web UI，适合内部演示、调试和产品走查。

- **本地配置和凭证管理**  
  通过 onboard、`~/.lampgo/config.toml`、`~/.lampgo/credentials.json` 管理模型、硬件和插件配置，避免把密钥写入仓库。

### 9.3 可替换或回退能力

- **TTS 回退**：MiMo TTS 不可用时，可使用 Edge TTS。
- **硬件回退**：没有真实硬件时，可使用 no-hw 模式调试 Web UI、路由和 Agent 流程。
- **模型回退**：配置层支持 openai、anthropic、gemini、local 等 provider，便于在不同环境下替换模型。

- 如果进入量产或大规模试用，需要补充设备安全认证、隐私策略、日志策略和远程运维方案。

## 11. 对内沟通口径

### 30 秒版本

lampgo 是一个桌面具身智能台灯机器人运行时。它把 LeLamp 这类 5 自由度机械台灯接入 MiMo 多模态模型、语音、摄像头、LED 表情和 OpenClaw Agent 生态，让用户可以通过自然语言让台灯看、听、说、动，并把复杂动作和交互沉淀成可复用技能。

### 1 分钟版本

lampgo 解决的是“AI 如何可靠驱动物理桌面设备”的问题。底层它提供安全的电机控制、平滑运动、急停、录制回放和 LED 表情；中层它把动作、表情、摄像头、语音封装成技能；上层它通过关键词、MiMo 多模态 Agent 和 OpenClaw 复杂任务链路，把自然语言和真实硬件连接起来。它既可以作为桌面 AI 伙伴 Demo，也可以作为小米内部探索具身智能、Agent 硬件协同和智能台灯产品形态的研发平台。

### 对外可包装关键词

- 桌面具身智能
- AI 台灯机器人
- 多模态智能硬件
- 有身体反馈的 AI 助手
- 可录制、可学习、可扩展的动作系统
- Agent 驱动的硬件执行端
- MiMo 多模态模型落地样板
